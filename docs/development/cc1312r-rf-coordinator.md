# CC1312R Sub-1GHz RF Coordinator <!-- trunk-ignore(markdownlint/MD025) -->

## Overview

This document describes the architecture and implementation of a CC1312R-based
sub-1GHz RF coordinator attached to the M5Stack Unit PoE P4 (ESP32-P4). It covers
why this hardware combination was chosen, how the two devices are wired, what
protocol runs between them, and how the ESP32-P4 driver (`cc1312_manager.h`) works.

The goal is to extend the sensor hub from a single-device station toward a multi-node
topology where remote CC1312R sensor nodes transmit readings over 868/915 MHz RF. A
coordinator CC1312R (on a LaunchPad XL) receives those RF packets and relays them over
SPI to the ESP32-P4, which publishes to MQTT over Ethernet.

Remote nodes can carry any sensor — PIR, door contact, LD2450 mmWave radar, time-of-flight
ranger, temperature probe, or any future type. The protocol is designed to accommodate
all of these without requiring changes to the framing or the ESP32-P4 driver.

## Current Status

The SPI link is fully operational. Confirmed working:

- Coordinator heartbeats received and parsed (`AA 0D 06 ...`)
- Node sensor readings received and published to MQTT (`cc1312/nodes`)
- Downlink node-list sync working — `acceptedNodeCount > 0`, `whitelistReady = 1`
- PIR trigger events (`0x10`) confirmed end-to-end
- ESP32-P4 parsing/pending-slot logic updated so trigger (`0x10`) and dwell (`0x11`)
  are treated as separate event types

**Clarification — PIR dwell ownership (2026-03-26):** Dwell is generated by the remote
node, not by the coordinator. The coordinator recognizes `0x11` dwell telemetry and
forwards it to the ESP32-P4 as a normal `SENSOR_READING` frame, using the same path as
`0x10` trigger telemetry. If no dwell frame appears on SPI, the likely causes are that
the node did not emit `RF_LINK_TELEMETRY_PIR_DWELL`, the running node build is older, or
the frame never reached the coordinator over RF. The coordinator does not contain a
separate dwell state machine or dwell synthesis step.

See [cc1312r-uart-to-spi-migration.md](cc1312r-uart-to-spi-migration.md) for the full
bring-up history and the technical rationale for each design decision.

## Network Topology

```text
[Remote node A]        [Remote node B]       [Remote node C]
CC1312R + PIR          CC1312R + LD2450      CC1312R + ToF + temp
        |                      |                     |
        +---------------868/915 MHz RF---------------+
                               |
                         [Unit PoE P4]
               CC1312R LaunchPad XL (coordinator)
                               |
                         SPI (G8–G12)
                 MOSI/MISO/CLK/CS/DRDY
                               |
                         ESP32-P4 host
                               |
                         RMII Ethernet
                               |
                          MQTT broker
```

Each remote node is a CC1312R with one or more attached sensors. The coordinator
CC1312R on the LaunchPad XL receives RF packets from all nodes, decodes them, and
forwards structured binary frames over SPI to the ESP32-P4.

## Interface: SPI

The initial PoC used UART. SPI was adopted after bring-up because:

- **Deterministic framing** — CS assertion provides a hard packet boundary without
  requiring a start-byte search in the driver.
- **DRDY handshake** — the coordinator asserts a DRDY GPIO only when a frame is
  primed in the SSI TX FIFO; the ESP32-P4 reads on the falling edge, eliminating
  the need for continuous RX polling.
- **Shared bus** — G8–G12 on the Hat2-Bus form a clean contiguous block, and SPI
  allows future peripherals to share the bus if needed.

See the migration document for the full trade-off analysis and bring-up notes.

## Hardware — Wiring

The CC1312R LaunchPad XL connects to the Unit PoE P4 via the **Hat2-Bus** (2.54mm
16P) connector.

| Signal             | Hat2-Bus | ESP32-P4 GPIO | CC1312R DIO | Notes                         |
| ------------------ | -------- | ------------- | ----------- | ----------------------------- |
| MOSI               | G8       | GPIO 8        | DIO11       | ESP32-P4 → CC1312R SSI_RX     |
| MISO               | G9       | GPIO 9        | DIO9        | CC1312R SSI_TX → ESP32-P4     |
| CLK                | G10      | GPIO 10       | DIO10       | Master clock, ESP32-P4 drives |
| CS                 | G11      | GPIO 11       | DIO8        | Active-low chip select        |
| DRDY               | G12      | GPIO 12       | DIO12       | CC1312R asserts low = ready   |
| Power              | 3V3      | —             | —           | 3.3 V logic rail              |
| Ground             | GND      | —             | —           | Common ground                 |

MOSI/MISO naming is from the master (ESP32-P4) perspective. In TI SysConfig, the
master MOSI connects to the slave SSI_RX; the master MISO connects to the slave SSI_TX.

## Frame Protocol

The binary frame format is unchanged from the original UART PoC:

```text
[0xAA] [LEN] [MSG_TYPE] [NODE_ADDR × 8] [RSSI] [...payload...] [CRC8]
```

| Byte      | Field     | Description                                                      |
| --------- | --------- | ---------------------------------------------------------------- |
| 0         | Start     | Always `0xAA`                                                    |
| 1         | LEN       | Byte count of MSG_TYPE + NODE_ADDR + RSSI + payload (excl. CRC) |
| 2         | MSG_TYPE  | Message type (see below)                                         |
| 3–10      | NODE_ADDR | 64-bit node address, big-endian                                  |
| 11        | RSSI      | int8, RF signal strength in dBm at the coordinator              |
| 12…(1+LEN)| Payload   | Message-type-specific bytes (see below)                          |
| 2+LEN     | CRC8      | Dallas/Maxim CRC8 over [MSG_TYPE + NODE_ADDR + RSSI + payload]  |

NODE_ADDR is 8 bytes, big-endian (network byte order). All payload multi-byte values
are little-endian.

### SPI Transport Contract

1. The ESP32-P4 is the SPI master; the CC1312R is the SPI slave.
2. The ESP32-P4 always drives CS. The CC1312R never asserts CS.
3. DRDY low means a complete uplink frame is already primed in the CC1312R SSI TX FIFO.
4. On DRDY low, the ESP32-P4 asserts CS once, clocks the frame header (START + LEN),
   then reads exactly LEN+1 more bytes (payload + CRC), then deasserts CS.
5. Downlink frames are sent by asserting CS without waiting for DRDY.
6. All batch downlinks (e.g. node-list sync) are sent within a single CS assertion so
   the CC1312R receives them in one DMA transfer.
7. SPI mode: `SPI_POL0_PHA1` on the CC1312R / `SPI_MODE1` on the ESP32-P4. Frequency: 1 MHz.

### Message Types

The SPI link is bidirectional. Uplink frames travel CC1312R → ESP32-P4. Downlink
frames travel ESP32-P4 → CC1312R. Both directions use the same frame structure.

#### Uplink (CC1312R → ESP32-P4)

| MSG_TYPE | Name              | NODE_ADDR          | Payload after RSSI                      |
| -------- | ----------------- | ------------------ | --------------------------------------- |
| 0x01     | NODE_STATUS       | source node        | addr_low32(4) tx_count(4) [fw(3) sc(1)] |
| 0x02     | SENSOR_READING    | source node        | sensor_class(1) + sensor-specific bytes |
| 0x03     | SENSOR_EVENT      | source node        | sensor_class(1) + sensor-specific bytes |
| 0x04     | NODE_SEEN         | seen node          | none — discovery mode only              |
| 0x05     | NODE_LIST_REQUEST | coordinator addr   | none — CC1312R asking for its node list |
| 0x06     | HEARTBEAT         | coordinator addr   | fw_major(1) fw_minor(1) fw_patch(1)     |
| 0x07     | PONG              | coordinator addr   | optional echoed payload                 |
| 0x24     | CONFIG_RESPONSE   | source node        | result(1) domain(1) param(1) flags(1) value_LE(4) |

NODE_LIST_REQUEST is sent on boot and every 10 s while the accepted list is empty,
then every 300 s as a keepalive once populated.

HEARTBEAT is sent every 30 s so the ESP32-P4 can track coordinator liveness
independently of node traffic.

#### Downlink (ESP32-P4 → CC1312R)

RSSI is always `0x00` in downlink frames.

| MSG_TYPE | Name                | NODE_ADDR      | Payload |
| -------- | ------------------- | -------------- | ------- |
| 0x10     | CMD_NODE_LIST_ENTRY | accepted node  | none    |
| 0x11     | CMD_NODE_LIST_END   | 0 (ignored)    | none    |
| 0x12     | CMD_ACCEPT_NODE     | node to add    | none    |
| 0x13     | CMD_REMOVE_NODE     | node to remove | none    |
| 0x14     | CMD_DISCOVERY_ON    | 0 (ignored)    | none    |
| 0x15     | CMD_DISCOVERY_OFF   | 0 (ignored)    | none    |
| 0x16     | CMD_PING            | echoed in pong | optional payload |
| 0x20     | CMD_GET_STATUS      | target node    | none    |
| 0x22     | CMD_GET_CONFIG      | target node    | domain(1) param_id(1) |
| 0x23     | CMD_SET_CONFIG      | target node    | domain(1) param_id(1) value_LE(4) |
| 0x25     | CMD_RESET_CONFIG    | target node    | domain(1) param_id_or_FF(1) |

In response to NODE_LIST_REQUEST, the ESP32-P4 sends all CMD_NODE_LIST_ENTRY frames
followed by CMD_NODE_LIST_END **in a single CS assertion** (batch send). This is
required because the CC1312R arms exactly one SPI DMA RX transfer per
`serviceSpiTransport()` call.

`CMD_GET_CONFIG`, `CMD_SET_CONFIG`, and `CMD_RESET_CONFIG` are currently
unicast-only commands. The coordinator ignores them if `NODE_ADDR` is the
broadcast address `FFFFFFFFFFFFFFFF`.

Like `CMD_GET_STATUS`, config commands are deferred until the coordinator next
hears telemetry from the target node. The coordinator then transmits the RF
request inside that node's short post-telemetry RX window.

### CONFIG_RESPONSE Payload

| Bytes | Field    | Type   | Description |
| ----- | -------- | ------ | ----------- |
| 0     | result   | uint8  | `0x00` success, otherwise error code |
| 1     | domain   | uint8  | Config domain ID |
| 2     | param_id | uint8  | Parameter ID, or `0xFF` for whole-domain reset |
| 3     | flags    | uint8  | Bitfield: `0x01` supported, `0x02` current value equals default, `0x04` restart required, `0x08` persisted to non-volatile storage |
| 4–7   | value    | uint32 | Applied/current value, little-endian |

This uplink is both the normal response to `GET_CONFIG` and the ACK for
`SET_CONFIG` / `RESET_CONFIG`.

Current result codes are:

- `0x00` = success
- `0x01` = unsupported domain
- `0x02` = unsupported param
- `0x03` = out of range
- `0x04` = invalid length

### NODE_STATUS Payload

| Bytes | Field       | Type   | Description                          |
| ----- | ----------- | ------ | ------------------------------------ |
| 0–3   | addr_low32  | uint32 | Low 32 bits of node IEEE address, LE |
| 4–7   | tx_count    | uint32 | Total packets transmitted since boot |
| 8     | fw_major    | uint8  | Node firmware major version          |
| 9     | fw_minor    | uint8  | Node firmware minor version          |
| 10    | fw_patch    | uint8  | Node firmware patch version          |
| 11    | sensor_type | uint8  | Sensor class ID (see registry below) |

Bytes 8–11 are optional (present when `LEN ≥ 12`). Older nodes may omit them.

### Sensor Class Registry

The first byte of a SENSOR_READING or SENSOR_EVENT payload identifies the sensor.

| Sensor class   | ID   |
| -------------- | ---- |
| PIR            | 0x01 |
| LD2450         | 0x02 |
| Time-of-flight | 0x03 |
| Door / state   | 0x04 |
| Temperature    | 0x05 |
| Raw / other    | 0xFF |

### Sensor-Specific Payloads

#### PIR (0x01) — Phase 2 format

The coordinator forwards the raw RF payload with no sensor-class prefix. The first
byte is the event type.

| body[0] | Event type | Remaining bytes                                           |
| ------- | ---------- | --------------------------------------------------------- |
| 0x10    | Trigger    | trigger_kind(1) event_count_LE(4) flags(1)                |
| 0x11    | Dwell      | dwell_seconds_LE(2) event_count_LE(4)                     |

`trigger_kind`: 0x01 = single detection, 0x02 = dual detection.

Coordinator behavior for PIR telemetry is intentionally simple:

- It validates `0x10` and `0x11` payload lengths.
- It repacks the decoded payload into the SPI/UART `SENSOR_READING` frame.
- It does not create, delay, or transform dwell events beyond that decode/forward step.

This means the presence of a dwell frame on SPI is direct evidence that the remote node
emitted `RF_LINK_TELEMETRY_PIR_DWELL` and that the coordinator received it over RF.

#### LD2450 (0x02)

| Bytes | Field     | Type    | Description                           |
| ----- | --------- | ------- | ------------------------------------- |
| 0     | n_targets | uint8   | Number of targets (0–3)               |
| 1–6   | Target 0  | 3×int16 | x_mm, y_mm, speed_cms (little-endian) |
| 7–12  | Target 1  | 3×int16 | (if n_targets ≥ 2)                    |
| 13–18 | Target 2  | 3×int16 | (if n_targets = 3)                    |

#### Time-of-Flight (0x03)

SENSOR_READING: `distance_mm(2 LE) quality(1)`
SENSOR_EVENT: `presence(1)` — 0 = clear, 1 = occupied

#### Door / State (0x04)

`state(1)` — 0 = closed, 1 = open

#### Temperature (0x05)

`temp_cdeg(4 LE)` — int32, temperature in centi-degrees (°C × 100)

#### Raw (0xFF)

Arbitrary bytes, published as a hex string in the `data` JSON field.

### CRC8

Dallas/Maxim CRC8 (polynomial `0x8C`, bit-reversed). Computed over
`[MSG_TYPE + NODE_ADDR + RSSI + payload]`. Start byte and LEN are excluded.

## Node Discovery and Enrollment

The ESP32-P4 NVS is the source of truth for the enrolled node list. The coordinator
holds only a RAM copy, populated on demand.

### Normal Operation

```text
CC1312R boots
  → sends NODE_LIST_REQUEST
  → ESP32-P4 waits 80 ms then replies:
      CMD_NODE_LIST_ENTRY × N  ]
      CMD_NODE_LIST_END        ]  single CS assertion (batch)
  → CC1312R: loads accepted list, begins forwarding sensor data

CC1312R re-sends NODE_LIST_REQUEST every 10 s while accepted list is empty
  → once populated, refreshes every 300 s and keeps sending HEARTBEAT every 30 s
```

### Discovering New Nodes

```text
MQTT: {"action": "discovery_on"}
  → ESP32-P4: sends CMD_DISCOVERY_ON to CC1312R
  → CC1312R: enters discovery mode — forwards all RF traffic, reports new addrs

New node transmits
  → CC1312R: sends NODE_SEEN (addr + rssi) to ESP32-P4
  → ESP32-P4: publishes to cc1312/seen, shows on /cc1312 web page

User clicks Accept (web page) or MQTT command:
  {"action": "accept_node", "addr": "00124B002D6D5A04"}
  → ESP32-P4: promotes addr to enrolled in NVS, sends CMD_ACCEPT_NODE to CC1312R
              then sends full batch node-list sync
  → CC1312R: adds to accepted list, begins forwarding sensor data

MQTT: {"action": "discovery_off"}
  → ESP32-P4: sends CMD_DISCOVERY_OFF to CC1312R
  → CC1312R: returns to normal mode — accepted nodes only
```

### Remote Management via MQTT

```text
{"action": "remove_node",   "addr": "..."}  — remove from NVS + CMD_REMOVE_NODE
{"action": "sync_node_list"}                — re-push full NVS list to CC1312R
{"action": "get_node_list"}                 — publish enrolled list to cc1312/config
{"action": "ping"}                          — send CMD_PING, log RTT from PONG
{"action": "get_status", "addr": "..."}     — request status from a specific node
```

## MQTT Topics

```text
cc1312/nodes    — sensor data from accepted nodes, JSON snapshot every 10 s
cc1312/seen     — nodes seen during discovery (pending approval), published on change
cc1312/config   — enrolled node list, published in response to get_node_list command
cc1312/command  — inbound commands (subscribed by ESP32-P4)
```

### `cc1312/nodes` Payload

```json
{
  "timestamp": 2724676,
  "coordinator_alive": true,
  "heartbeat_age_ms": 18211,
  "coordinator_fw": "0.1.0",
  "messages": [
    {
      "node": "00124B002D6D5A04",
      "msg": "reading",
      "rssi_dbm": -34,
      "age_ms": 122,
      "sensor": "pir",
      "event": "dwell",
      "dwell_seconds": 7,
      "event_count": 176
    }
  ]
}
```

All pending messages are upserted by `(node_addr, msg_type, sensor_class)`. For PIR
events, `data[0]` (the event subtype: `0x10` trigger or `0x11` dwell) is also included
in the key so that trigger and dwell occupy separate slots and are both published in the
same batch. For all other sensor classes, if the same node sends the same type and
sensor class multiple times between publishes, only the most recent reading is retained.

## ESP32-P4 Driver — `src/cc1312_manager.h`

Header-only class. Constructor takes `MQTTManager&`. No `.cpp` file.

- `begin()` — initialises the SPI bus and sets CS high, DRDY as input.
- `update()` — called every loop iteration; detects DRDY falling edge, reads one
  frame per edge, runs the parser, fires deferred downlinks, publishes on the 10 s
  interval.
- `handleCommand(action, doc)` — dispatches MQTT commands (accept_node, remove_node,
  discovery_on/off, sync_node_list, get_node_list, ping, get_status).
- `isCoordinatorAlive()` — returns true if a HEARTBEAT was received within 90 s.

### Frame Parser

`_parseByte()` is a byte-at-a-time state machine:

1. Wait for `0xAA` start byte — enter frame mode.
2. Receive LEN — abort if LEN > `CC1312_MAX_PAYLOAD` (64).
3. Accumulate until `start(1) + LEN(1) + LEN + CRC(1)` bytes are buffered.
4. Validate CRC8 over `[MSG_TYPE + NODE_ADDR + RSSI + payload]`.
5. Dispatch to `_dispatchFrame()`.

### Diagnostic Logging

Every 5 seconds (only printed when there is activity or after 35 s silence):

```text
[CC1312] DRDY triggers=1 errors=0 (last bad=0x00) bytes=20 pending=0
[CC1312] No data in 35s — check DRDY (G12), CS (G11), SPI wiring and 3V3
```

Per decoded frame:

```text
[CC1312] Coordinator heartbeat (id=00124B001CAA4B8F) fw=0.1.0
[CC1312] 00124B002D6D5A04 reading/pir len=7 (rssi=-34)
[CC1312] RAW(20): AA 11 02 00 12 4B 00 2D 6D 5A 04 DE 11 07 00 B0 00 00 00 D6
```

## Enabling and Verifying

### Step 1 — Flash the CC1312R Coordinator Firmware

Flash the LaunchPad XL with coordinator firmware (`rfEchoRx.c`). The firmware:

1. Receives RF packets via a proprietary 868/915 MHz PHY.
2. Filters packets against the accepted-node whitelist.
3. Relays decoded frames to the ESP32-P4 over SPI slave (SSI, `SPI_POL0_PHA1`).
4. Responds to downlink commands (node-list sync, discovery, ping, get-status).

### Step 2 — Wire Up and Enable

Connect the Hat2-Bus as per the wiring table above. Set `ENABLE_CC1312=1` in
`platformio.ini` for the `m5tab5-esp32p4` environment and flash.

Expected boot log:

```text
[CC1312] Initialized on SPI MOSI=8 MISO=9 CLK=10 CS=11 DRDY=12 @ 1000000Hz
[CC1312] Loaded N enrolled nodes from NVS
```


## Known Issues

### PIR Dwell Framing Compatibility (resolved 2026-03-26)

**Symptom:** PIR trigger frames (`body[0]=0x10`) were visible on SPI, but PIR dwell
frames (`body[0]=0x11`) were missing even when the node-side test configuration was
expected to emit a fixed dwell after a dual trigger.

**Root cause:** The running CC1310 node was emitting an `8`-byte dwell payload variant
while the coordinator decoder in this workspace only accepted the `7`-byte dwell shape.
The coordinator therefore received dwell on RF but rejected it before host forwarding.

**Fix:** `rfCoordinator/rfEchoRx.c` now accepts both dwell payload variants:

- `7` bytes — `event_type(1) + dwell_seconds_LE(2) + event_count_LE(4)`
- `8` bytes — same fields plus a trailing dwell `flags(1)` byte

The coordinator forwards the canonical `7`-byte PIR dwell body to SPI/ESP32-P4 after
decode, ignoring the extra trailing flags byte when present.

**Observed result after fix:**

- dual trigger arrives as normal `reading/pir`
- dwell arrives as normal `reading/pir`
- MQTT publishes both independently, for example:
  - `event="trigger"`, `trigger_kind="dual"`, `event_count=49`
  - `event="dwell"`, `dwell_seconds=2`, `event_count=50`

### Temporary Dwell Debug Frames (2026-03-26)

During bring-up, the coordinator emitted temporary debug frames so the CC1312 and
ESP32 teams could determine where dwell was being lost.

These frames are carried as normal `SENSOR_READING` uplinks with sensor class
`Raw / other (0xFF)` so the ESP32 can relay them without needing a new message type.

#### Current temporary coordinator debug frame

- `MSG_TYPE = 0x02` (`SENSOR_READING`)
- `NODE_ADDR = coordinator address`
- `RSSI = 0`
- `payload[0] = 0xFF` to force `reading/raw`
- payload length = `13`

Payload layout:

```text
byte 0:  0xFF raw/debug marker
byte 1:  dual_trigger_count[0]   little-endian
byte 2:  dual_trigger_count[1]
byte 3:  dwell_seen_count[0]
byte 4:  dwell_seen_count[1]
byte 5:  dwell_reject_count[0]
byte 6:  dwell_reject_count[1]
byte 7:  dwell_queued_count[0]
byte 8:  dwell_queued_count[1]
byte 9:  dwell_queue_drop_count[0]
byte 10: dwell_queue_drop_count[1]
byte 11: dwell_forwarded_count[0]
byte 12: dwell_forwarded_count[1]
```

Interpretation:

- `dual > 0`, `seen = 0` → no dwell-like RF telemetry reached the coordinator
- `seen > 0`, `reject > 0` → coordinator received dwell but rejected payload shape
- `queued > 0`, `forwarded = 0` → coordinator accepted dwell but did not emit it to SPI
- `forwarded > 0` → dwell left the coordinator; any remaining bug is downstream

Example from a successful post-fix capture:

```text
data="010001000000010000000100"
```

Decoded:

- `dual=1`
- `seen=1`
- `reject=0`
- `queued=1`
- `queue_drop=0`
